# CLAUDE.md — Ring of Fire (1994) digitization project

Project memory and working rules for Claude Code. Read this before editing anything.

## What this project is

A faithful, accessible, modern digital edition of **"Ring of Fire: The Handgun Makers of
Southern California"** — a 112-page scanned 1994 public-health policy report by
Garen J. Wintemute (UC Davis Violence Prevention Research Program). The original is a
print-locked scanned PDF with no text layer.

The work converts it into clean, portable, **WCAG 2.2 AA** deliverables: a structured
Markdown source, a single self-contained accessible HTML edition, cropped figure images
with descriptive alt text, a Mermaid diagram of the family tree, and a chart toolchain
that turns the report's data into accessible interactive charts.

This is a digitization/accessibility project. It is legitimate public-health/policy
content; there is nothing weapon-enabling here. The job is fidelity + accessibility,
not analysis or opinion.

## NON-NEGOTIABLE rules

1. **Faithful transcription. Never fabricate or "update" data.** The text and figures
   must match the 1994 original. Do not modernize statistics, add post-1994 facts, or
   "correct" the report's conclusions. If asked to "update it," that means cleaner
   formatting/structure, NOT changed content.
2. **Preserve original errors as printed**, with a marker where helpful. Known examples:
   the Sundance A-25 row in Table 2-14 prints a markup of "89" for a $45→$80 step (an
   original discrepancy, kept as-is); "(sic)" markers; naming like "LHC-380" vs "LH-380"
   kept as printed.
3. **Chart data must come from real sources, never eyeballed off a printed chart.**
   The exact integer datasets we have (see `charts/data/`) come from labeled tables/figures.
   The yearly *production-trend line charts* (Figures 2-1, 2-5, 2-6, 2-9, 2-12, 2-15, 2-18,
   2-21) have NO exact source numbers in hand — do not invent per-year values. Digitize them
   from the BATF/DOJ/ATF sources the report cites, or leave them as the existing cropped
   images, and say so plainly.
4. **Accessibility is a hard requirement, not a finish.** Every image has descriptive,
   non-empty `alt`. Every data chart ships the layered pattern below. No color-only encoding.
   `lang="en"`. Real table markup with `<caption>`, `scope`. Keep contrast adequate.

## Accessibility conventions (apply everywhere)

- **Alt text:** concise and content-bearing on the `<img>`; put the *data* in the visible
  caption / long description, not crammed into alt. Charts get a one-line identifying alt;
  the numbers live in the caption + data table.
- **The layered chart pattern (standardized):** every data chart provides the same
  information three independent ways — (a) the interactive chart with keyboard navigation
  and per-point screen-reader labels, (b) audio sonification, and (c) a real `<table>` text
  alternative that works with scripting disabled. `charts/generate_charts.py` produces this.
- **Self-contained HTML is preferred** for deliverables: inline everything (CSS, JS, images
  as base64) so a file works offline and in sandboxed viewers. A file that depends on a CDN
  is NOT self-contained — that was the original Highcharts bug; do not reintroduce it.
- **Figures → `<figure>`/`<figcaption>`**; formulas exposed with `role="math"` + `aria-label`.

## Build / run

```bash
pip install -r requirements.txt        # markdown + weasyprint (+ altair/vl-convert/plotly for charts)

# Export targets — choose the deliverable format(s):
make md         # assemble sections -> document/Ring-of-Fire-1994.md  (Markdown)
make html       # -> self-contained accessible HTML (alias: make doc)
make pdf         # -> tagged accessible PDF/UA-1 (built from the HTML)
make export     # build md + html + pdf in one pass
make charts     # regenerate every standalone chart in charts/charts.json into charts/out/
make charts-static  # regenerate the static SVG fallbacks (charts/static/) embedded in HTML/PDF
make diagrams   # render the Mermaid diagrams (Figure 1-1 family tree) to PDF-safe SVG
```

`make html` and `make pdf` depend on `charts-static` and `diagrams`, so the embedded chart
figures and the Mermaid diagram(s) are always rebuilt before each edition.

Each format target re-runs `assemble.py` first, so a single `make pdf`/`make html`/`make md`
always reflects the current `build/` sources. **Never edit `document/Ring-of-Fire-1994.md`
directly** — it is overwritten by `assemble.py`; edit the section files in `build/`.

**Accessible PDF (`build_pdf.py`):** renders the already-accessible HTML edition through
WeasyPrint with `pdf_variant="pdf/ua-1"`. The HTML's semantics carry straight into the PDF
tag tree (headings, `<table>`/`<th scope>`, `<figure>` with per-image `/Alt`, list and
blockquote tags), and the variant enforces PDF/UA-1: `/MarkInfo /Marked true`,
`/StructTreeRoot`, `/Lang en`, `DisplayDocTitle true`, `pdfuaid:part=1`, plus a print
stylesheet for page setup and heading bookmarks. Because the PDF is derived from the HTML,
its accessibility tracks the HTML's — fix issues upstream in `build/` / `build_html.py`.
WeasyPrint needs system Pango/cairo/gdk-pixbuf (already present here).

## Charts — decision record and how-to

We compared five approaches for accessible interactive charts (artifacts in
`charts/comparison/`). Findings:

- **Highcharts + accessibility module = the chosen approach.** Best built-in keyboard
  nav, screen-reader output, and sonification; exports PNG/JPEG/SVG/PDF client-side.
  Must be **inlined** (we vendor it in `charts/lib/`) to stay self-contained/offline.
  License: free for personal/non-commercial use — verify before any commercial use.
- **Plotly** — richest interactivity and tested well with AT, but its native a11y leans on
  the visual SVG (we add a table) and **static PDF export needs headless Chrome via Kaleido**
  (`plotly_get_chrome`). Good for interactive web, weak for scripted PDF.
- **Altair / Vega-Lite + vl-convert** — best for **scripted, browserless vector export**
  (PDF/SVG/PNG) for print/Word/LaTeX pipelines. Use this when you need static vector output.
- **Hand-built SVG** (`comparison/handbuilt-svg-demo.html`) — zero-dependency reference of
  the layered pattern.

Generate one: `cd charts && python3 generate_charts.py data/figure-2-2_caliber-share.csv --title "…" --ymax 100 --out out/figure-2-2.html`
Generate all: `cd charts && python3 generate_charts.py --manifest charts.json`

CSV format: column 1 = category (x), remaining columns = one or more numeric series
(header = series name). Single series → bar; multiple → grouped bars.

**Interactivity never survives into PDF** — a PDF is a static snapshot. For PDF/print,
export a vector image (Highcharts menu, or vl-convert) AND pair it with a tagged data
table + alt text in the tagged PDF (PDF/UA). Don't expect one library to do both.

### Charts embedded in the HTML/PDF editions (the `<!--CHART id-->` pipeline)

Four exact-data charts are embedded directly in the deliverables next to (never replacing)
the faithful cropped scans/tables: **Figure 1-1** (1992 production), **Figure 2-2** (caliber
share), **Figure 2-4** (relative stopping power), and **Table 4-2** (tracing vs production).

- **Source of truth = the CSVs in `charts/data/`.** No markdown chart tables to drift.
- **Placement = an HTML comment token** in the `build/` source: `<!--CHART figure-1-1-->`
  on its own line, right after the figure's caption/table. It is invisible in the Markdown
  deliverable and is expanded only by `build_html.py`. The token `id` is the CSV/out filename
  stem (matched via `embed.manifest_by_id()`).
- **One figure, three representations (the layered pattern), works in both editions:**
  `charts/embed.py` emits a `<figure>` containing (a) an **interactive Highcharts** div
  (`.chart-interactive`, shown on screen, empty in print), (b) a **sonification** button,
  (c) a **static SVG** `<img>` (`.chart-static`, base64-inlined, hidden on screen, shown in
  print — this is what gets a `/Figure` + `/Alt` tag in the PDF), and (d) a real **data
  `<table>`** (always shown). Print CSS in `build_html.py` toggles interactive↔static; since
  WeasyPrint applies print media and runs no JS, the interactive divs contribute nothing to
  the PDF while the static SVG + table carry the data.
- **Only the Highcharts core + accessibility + sonification modules are inlined** for embedding
  (`embed.EMBED_LIBS`); exporting/jspdf/svg2pdf are skipped to keep the page lean. The static
  SVGs come from `charts/generate_static.py` (Altair/vl-convert, browserless).
- **To embed another chart:** add its CSV + a `charts.json` entry, run `make charts-static`,
  then drop a `<!--CHART that-id-->` token after its caption in the relevant `build/` file.

### Diagrams — the Mermaid `<!--MERMAID id-->` pipeline (Figure 1-1 family tree)

The Figure 1-1 family tree is a *workflow/relationship* diagram, not tabular data, so it is
authored as Mermaid (`charts/family-tree.mmd`) and rendered to a PDF-safe SVG. It is embedded
**next to** (never replacing) the faithful cropped scan and the data table.

- **Placement = an HTML comment token** in the `build/` source: `<!--MERMAID family-tree-->`
  on its own line (in `build/01-snsfamily.md`, after the table, before the `<!--CHART-->`).
  Invisible in the Markdown deliverable; expanded only by `build_html.py`. The token `id` is
  the `.mmd`/`.svg` filename stem.
- **Render: `make diagrams` → `charts/render_diagrams.py`** runs `mmdc` (mermaid-cli) with
  `charts/mermaid.config.json` + `charts/puppeteer.config.json`, writing `charts/static/family-tree.svg`.
  `mmdc` is an **npm** package (not pip): `npm i -g @mermaid-js/mermaid-cli`, and it needs a
  headless Chromium (e.g. `npx puppeteer browsers install chrome-headless-shell`). The script
  locates Chrome via `PUPPETEER_EXECUTABLE_PATH` → `~/.cache/puppeteer` → system Chrome, and
  **skips gracefully** if `mmdc`/Chrome are absent (the committed SVG is reused → offline builds work).
- **CRITICAL render constraint — root-level `"htmlLabels": false`.** WeasyPrint cannot draw SVG
  `<foreignObject>` and runs no JS, so foreignObject labels render blank in the PDF. Forcing
  native `<text>`/`<tspan>` requires `"htmlLabels": false` at the **root** of the config; the
  scoped `flowchart.htmlLabels` key alone does NOT take effect. Verify the rendered SVG has zero
  `<foreignObject>`.
- **Mermaid 11 gotcha — no `color:` on any `classDef`.** Under htmlLabels:false, a per-class
  `color:` property makes MULTI-ROW label text (wrapped or `<br/>`) render invisibly on all
  nodes. Leave label color to the theme default (`primaryTextColor:#1a1a1a`) and distinguish
  the origin node with a light fill + thick accent border. Also avoid a literal `&` in labels
  (it double-encodes to `&amp;amp;`) — use "and".
- **One figure, layered, both editions:** `charts/embed.build_mermaid_html()` emits a `<figure>`
  with the base64-inlined SVG as an inert `<img>` (an `<img>` runs no SVG script; it gets a
  `/Figure` + `/Alt` tag in the PDF), a figcaption carrying the `.mmd`'s `accDescr` long
  description, and the adjacent data table as the tabular alternative. The `.mmd`'s `accTitle`/
  `accDescr` are single-sourced (they also become the SVG `<title>`/`<desc>`).

## Figure / OCR notes (gotchas)

- The PDF is fully **scanned** — each page is one raster image, so figures are not separately
  embedded; they were cropped from 200-DPI page renders (`scripts/crop_figures.py`).
- **OCR garbles tables and figures.** Prose OCR is good; any table/figure values were
  transcribed by viewing the page image, then sanity-checked against internal totals/markups.
- **Figure 2-14** (Davis pistol): the original printed caption is too faint on the scan to read;
  identified from the "Davis Industries, Chino, CA" slide engraving. Noted in its caption.
- **"Figure 5-1" does not exist as an image.** The body text says "Figure 5-1" but the actual
  exhibit is printed as **Table 5-1** (BATF factoring criteria) on pp. 70–71, rendered as an
  accessible data table. Don't go hunting for a 5-1 image.
- The family tree (Figure 1-1) exists four ways: the original cropped scan, a Markdown table in
  `build/01-snsfamily.md`, the Mermaid source `charts/family-tree.mmd`, and its rendered
  accessible SVG `charts/static/family-tree.svg` (embedded via `<!--MERMAID family-tree-->` —
  see "Diagrams" above). All are kept; the scan is never replaced.
